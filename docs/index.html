01

bug-fix-orchestrator

Задача

Паралельно виправити три незалежні, навмисно засіяні баги у трьох непересічних файлах продакшн-коду – за один виклик, без ручного перемикання між ними.

Результат

Оркестратор розбив задачу на три ізольовані worktree-воркери, кожен отримав свій баг і файл, усі три запрацювали паралельно. Кожен фікс верифіковано через type-check, після чого воркери зведені в один підсумковий звіт із diff'ами й посиланнями на worktree для рев'ю. 

const GRADIENT_BY_PRIORITY = {
-  low: 'danger',
+  low: 'success',
  medium: 'warning',
-  high: 'success',
+  high: 'danger',
};

Один із трьох diff'ів у звіті – інверсні кольори priority-бейджа, знайдено і виправлено воркером самостійно.

  • Priority badge colors – інверсні кольори low/high пріоритету
  • Subtask progress count – лічильник рахував незавершені підзадачі замість завершених
  • "Just now" threshold – поріг порівнювався в мілісекундах з константою в секундах
02

ai-qa-runner

Промпт

«Зроби ai-qa-runner аудит останніх змін»

Результат

Проаналізував три останні коміти й до злиття склав перелік ризиків для ручної перевірки: конфігурацію підпису iOS-збірки, стабільність патча сторонньої бібліотеки при апдейтах версій, сумісність із чистим npm ci. Кожен пункт – з конкретним кроком перевірки, а не загальним попередженням.

  • Signing config. Team ID у Xcode-проєкті прив'язаний до одного акаунта – позначено як пункт для перевірки на іншій машині/CI перед релізом.
  • Стабільність патча. Заміна рядка в сторонньому заголовку не має self-check – запропонована перевірка через grep після кожного pod install.
  • Lockfile noise. Виявив непов'язаний ~400-рядковий diff у package-lock.json всередині iOS-коміту – винесено окремим пунктом для чистішого рев'ю.
03

ai-search-indexer

Промпт

«Хочу додати пошук по записах логу – щоб користувач міг знайти запис за словом у заголовку чи описі, і в ідеалі за змістом (напр. "запис про тренування" знаходив записи про біг). Подивись як зараз зберігаються дані і запропонуй реалізацію.»

Результат

Розібрав поточну модель даних (AsyncStorage + Zustand, без бекенду) і запропонував двофазну архітектуру: миттєвий keyword-пошук уже зараз (Fuse.js, без нових бекенд-залежностей), і семантичний пошук другою чергою – з порівнянням двох реалістичних шляхів (on-device embeddings проти LLM-as-ranker на вже наявному BYOK Anthropic-ключі) та trade-off'ами по вартості, затримці й офлайн-роботі.

  • Фаза 1 – keyword. Fuse.js по title/description/notes, live-пошук без затримки, працює повністю офлайн.
  • Фаза 2 – семантика. Порівняно on-device embeddings (офлайн, +розмір застосунку) проти LLM-as-ranker (без нової залежності, +латентність на запит).
  • Правило деградації. Keyword-пошук лишається фолбеком завжди – семантика вмикається тільки якщо є ключ і мережа.
04

content-recommender-scaffold

Промпт

«Хочу додати блок "Рекомендовано для тебе" на головному екрані, based on історії записів користувача. Немає ще реальної ML-моделі – зроби скелет з мок-даними і чіткою межею для підключення бекенду пізніше.»

Результат

Реалізував робочий блок на головному екрані: інтерфейс рекомендації, хук useRecommendations() як єдина точка підключення реального бекенду в майбутньому, і мок-евристику на реальних полях запису (статус, пріоритет, час останньої активності, підзадачі) – без жодного вигаданого поля. UI відповідає наявній дизайн-системі застосунку.

  • Seam. UI викликає лише useRecommendations() – заміна мок-евристики на реальний бекенд не торкається екранів.
  • Евристика. Застояні in-progress, невиконані high-priority, записи з незавершеними підзадачами – кожен з поясненням чому показаний.
  • Аналітика. recommendation_shown/recommendation_tap вже інструментовані, готові під підключення реального sink.

05

product-tradeoff-challenger

Промпт

«Плануємо додати streak-механіку (серія днів поспіль з записами) з push-нагадуваннями. Розбери цю ідею як скептичний продуктовий інженер – які питання ми не задали, що може піти не так.»

Результат

Структурований продуктовий розбір: альтернативні підходи (зниження friction логування замість тиску на щоденність), питання для PM/дизайну перед білдом, конкретна метрика успіху замість самої лише довжини streak, і сигнали, за якими варто стежити після запуску.

  • Альтернатива. Знизити friction логування (quick-add, віджет) замість тиску на щоденність.
  • Питання перед білдом. Що саме рахувати "днем" – відкриття застосунку чи створення запису?
  • Метрика. 30-денний retention чи entries/week – а не сама довжина streak, яка тавтологічна.
06

retention-audit

Промпт

«Проаудитуй головний екран додатку на предмет retention-механік – чого не вистачає (empty states, push hooks, reward loops, streaks) і що можна додати.»

Результат

Наскрізний аудит головного екрана за empty states, re-engagement тригерами й reward loops – пріоритезований беклог із семи конкретних покращень, кожне з оцінкою складності реалізації.

  • Найвищий пріоритет. Тижнева passive-зведення на головному екрані – вже на наявних полях, без нового бекенду.
  • Швидкий win. Onboarding empty state для першого запуску замість статичного тексту.
  • Дешевий фікс. Позитивний fallback у рекомендаціях замість повного зникнення блока.
